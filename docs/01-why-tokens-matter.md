## 1.1 什麼是 Token？

Token 是大型語言模型閱讀和寫入的基本單位。它們不是單字，也不是字元——它們是**子詞 (subwords)**，由一種稱為位元組對編碼 (Byte-Pair Encoding, BPE) 的演算法建立。

### BPE 如何運作

1. 從原始位元組開始（256 個可能值）
2. 在訓練數據中找到最頻繁的位元組對
3. 將這些對合併為新的 Token
4. 重複直到詞彙量達到約 10 萬個 Token

結果：常見的英文單字變成單個 Token，而罕見或長的單字則被拆分成碎片。

### 範例

| 文本 | Token | 計數 |
|------|--------|-------|
| `hello` | `hello` | 1 |
| `unhappiness` | `un` + `happiness` | 2-3 |
| `authentication` | `authentication` | 2-3 |
| `I met a huge dog` | `I` `met` `a` `huge` `dog` | 5 |
| `Sure, I'd be happy to help you with that!` | (每個單字/標點符號) | ~10 |

最後一個例子——10 個 Token 的零資訊量。這就是 Token 優化背後的核心洞察。

**關鍵事實：** 在 Claude 3 分詞器中，10,000 個最常見的英文單字中有 8,311 個被標記為單個 Token。常見單字很便宜。填充詞仍然各佔 1 個 Token——而且它們沒有增加任何資訊。

## 1.2 為什麼你應該在乎

你發送或接收的每個 Token 都有成本。以下是其影響：

| 維度 | 影響 |
|-----------|--------|
| **成本** | Token = 金錢。API 定價是按 Token 計算的。GitHub Copilot 的使用限制是基於 Token 的。**關鍵的不對稱性：輸出 Token 的成本是輸入 Token 的 5 倍**（Anthropic：Haiku $1/$5，Sonnet $3/$15，Opus $5/$25 每百萬 Token）。Copilot UBB 使用相同的桶結構。這意味著回應中每個不必要的句子，其成本是你在提示詞中使用相同句子的 5 倍。輸出控制是本指南中所有技術中每 Token 投資報酬率 (ROI) 最高的 |
| **速率限制** | 每次請求的 Token 越少 = 在達到上限之前，每分鐘可以進行更多請求 |
| **上下文窗口** | 輸入（你的提示詞 + 上下文）和輸出（回應）共享的有限空間。浪費輸入 Token → 輸出的空間變少 |
| **速度** | 輸入 Token 越少 = 獲得第一個 Token 的時間越快。你實際上能更快地獲得答案 |
| **計費** | 自 2026 年 6 月 1 日起，GitHub Copilot 按 **用量計費 (UBB)** 計費。每次互動都在三個桶中計量——輸入 Token（提示詞 + 上下文 + 歷史記錄）、輸出 Token（回應）和快取 Token（重複使用的上下文，按輸入費率的約 10% 計費）。成本 = (輸入 × 輸入費率) + (快取 × 快取費率) + (輸出 × 輸出費率)。Business 和 Enterprise 席位包括在整個組織中共享的池化 AI 額度（CB 席位 $30，CE 席位 $70）。代碼補全保持無限使用 |

**複合效應：** 你的大部分輸入 Token 並非來自你輸入的內容——它們來自文件上下文、對話歷史、系統提示詞和 MCP 工具架構。在 Agent 模式下，你輸入的提示詞通常只佔總輸入（可達 5,000–50,000+）中的 5-100 個 Token。最大的輸入槓桿是結構性的（指令文件大小、分頁管理、上下文範圍）。最大的每 Token 成本槓桿是輸出：5 倍的定價溢價，可以通過 `copilot-instructions.md` 中的一行直接解決。

## 1.3 GitHub Copilot 如何使用 Token

了解 Copilot 在幕後做什麼有助於你進行優化：

```text
┌─────────────────────────────────────────────────┐
│                 上下文窗口 (Context Window)     │
│                                                 │
│  ┌──────────────────┐  ┌─────────────────────┐  │
│  │  輸入 TOKEN      │  │  輸出 TOKEN         │  │
│  │                  │  │                     │  │
│  │  系統提示詞      │  │  你接收到的         │  │
│  │  + copilot-      │  │  回應               │  │
│  │    instructions  │  │                     │  │
│  │  + 文件上下文    │  │                     │  │
│  │  + 對話歷史      │  │                     │  │
│  │  + 你的提示詞    │  │                     │  │
│  └──────────────────┘  └─────────────────────┘  │
└─────────────────────────────────────────────────┘
```

- **系統提示詞：** Copilot 自己的指令（你無法控制）
- **`copilot-instructions.md`：** 你的專案級指令——在**每次**互動時加載
- **文件上下文：** 你使用 `#file` 引用的文件，加上 Copilot 自動包含的文件（開啟的分頁、導入、附近的文件）
- **對話歷史：** 當前對話中的所有先前消息
- **你的提示詞：** 你實際輸入的內容

**隱藏開銷：** Copilot 會添加你看不見的上下文——存儲庫結構、文件內容、導入圖。這就是為什麼你 20 個字元的提示詞可能會消耗 2000+ 個輸入 Token。

**編碼代理 (Coding Agent) 乘數：** GitHub Coding Agent 執行多步驟會話。每一步都涉及閱讀文件、做出決定、編寫代碼。單個從問題到 PR 的會話可能會消耗數萬個 Token。節省的成本會在這些步驟中複合。

## 1.4 本指南適用於誰

- **GitHub Copilot Chat 用戶** —— 主要受眾。大多數技術都適用於此
- **GitHub Copilot Inline/Edits 用戶** —— 部分技術適用（提示詞控制有限）
- **GitHub Coding Agent 用戶** —— 高價值受眾。Token 節省在長會話中會複合
- **任何關注效率的人** —— 即使你沒有達到限制，更快的響應和更簡潔的互動也是值得的
