## 3.1 正面對比：相同提示詞，不同技術

**任務：** "Add error handling to this function" (為此函數添加錯誤處理)

| 技術 | 提示詞 | ~輸入 Token | 輸出品質 |
|-----------|--------|--------------|----------------|
| 冗長英文 | "Hey, could you please add comprehensive error handling to this function? I'd like it to handle all edge cases including null inputs, invalid types, and network errors. Please explain your changes." | ~40 | 好，但輸出冗長 |
| 原始人式 (輕量) | "Add error handling to this function. Cover null inputs, invalid types, network errors." | ~16 | 好 |
| 原始人式 (全面) | "Add error handling. Cover: null input, bad type, network error." | ~12 | 好 |
| 原始人式 (極限) | "Error handling: null/bad-type/net-err." | ~7 | 好 (可能需要上下文) |
| 結構化 | `fn: add error handling\n- null input\n- invalid type\n- network error` | ~12 | 好 |
| 以代碼為中心 | `# TODO: handle None, TypeError, ConnectionError` | ~8 | 好 |

這六種方式都能產生正確的錯誤處理代碼。Token 成本從 7 到 40 不等。對於相同的結果，成本相差 **5.7 倍**。

### 核心大贏家

如果你什麼都不做，至少做這八項。按影響力與難易度比率排名：

1. **原始人式語法 (Caveman-speak)** —— 節省 30-50% 輸入 Token；配合 B5 可節省 40-55% 輸出 Token
2. **精確提示詞** —— 節省 30-60%，只需改變習慣
3. **僅代碼 / 約束輸出** —— 節省 40-80% 輸出 Token，只需一條指令
4. **縮減始終開啟的上下文** (`copilot-instructions.md` + `AGENTS.md`) —— 壓縮填充詞、修剪為僅留地雷、刪除 LLM 生成的樣板。在每次互動和代理步驟中複合；節省 20-23% 代理任務成本並提升正確率
5. **用 Ask Mode 處理簡單問題** —— 通過避免代理開銷節省 60-90%
6. **審計 MCP 伺服器** —— 禁用未使用的伺服器，每個代理任務節省 5K-190K Token
7. **先將非文本文件轉為 Markdown** —— 在內容進入對話、代理或 RAG 之前，避免 DOCX/PDF/HTML 的格式稅
8. **針對目標模型調優提示詞** —— 非單次請求縮減，但能提高首試品質並避免返工

## 3.4 品質影響評估

壓縮會損害輸出品質嗎？研究表明：**極少發生，且僅在極端壓縮水平下發生。**

| 壓縮級別 | 品質影響 | 證據 |
|-------------------|---------------|----------|
| Lite (去除填充詞) | 無 | 模型在多樣化文本上訓練，能理解乾淨的散文 |
| Full (去除冠詞、碎片化) | 微乎其微 | 模型能很好地處理碎片化語句；技術術語得以保留 |
| Ultra (縮寫、箭頭) | 輕微風險 | 複雜的多步驟指令可能被誤讀 |
| Wenyan (文言文) | 中度風險 | 模型對文言文的理解不如英文可靠 |
| Extreme (僅單個單詞) | 顯著風險 | 歧義增加，模型可能誤解 |

**臨界點：** 當你發現自己需要重新解釋或得到錯誤結果時，說明你壓縮過度了。請往回退一個級別。

**特定模型註記：** 所有主流模型 (GPT-4, Claude, Gemini) 都能很好地處理「全面原始人式 (full caveman)」。對於熟悉該領域的資深使用者，可以使用「極限 (Ultra)」。文言文在代碼生成任務中不可靠。

### 收益遞減

節省曲線並非線性的。前 30% 的壓縮（去除填充詞）是免費的。接下來的 20%（碎片化、縮寫）也幾乎是免費的。超出此範圍後，每個額外的壓縮點都會帶來品質風險。

```text
節省 vs. 品質風險：

品質  ████████████████████████████████████░░░░░░░░░
風險  ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░███████████████
         0%       20%      40%      60%      80%
                      Token 節省 →
               lite     full      ultra    extreme
```

**最佳平衡點：全面原始人式 (full caveman)（節省 30-50% 輸入 Token；配合精簡系統指令可節省 40-55% 輸出 Token）。** 回報最大，風險極小。
