# 用於 Copilot 成本控制的企業治理

[← 返回指南](index.md)

---

本頁面面向組織和企業管理員。目標：在不發明 GitHub 未記錄的控制項的情況下控制支出。

## 什麼能真正控制支出

三個槓桿最重要：

1. 預算上限 (Budget ceilings)
2. 使用者級別的預算控制
3. 模型可用性

提示詞壓縮仍然重要，但它是一個使用效率槓桿。它不是管理員計費控制項。

有一個時間規則至關重要：2026 年 6 月 1 日是截止日期。在那之後，Business 和 Enterprise 治理將轉向用量計費下的 AI 額度使用量。請將進階請求視為遺留的過渡背景，而非主要的規劃模型。

## 1. 首先設置預算

請使用 GitHub 的 [針對組織和企業的用量計費指南](https://docs.github.com/en/copilot/concepts/billing/usage-based-billing-for-organizations-and-enterprises) 以及 [計費產品預算設置文檔](https://docs.github.com/en/billing/how-tos/budgets/setting-up-budgets-to-control-spending-on-metered-products)。

6 月 1 日之後 Business 和 Enterprise 的 GitHub 文檔關鍵點：

- 預算可以在組織、企業或成本中心範圍內設置
- 你可以啟用「達到預算時停止使用」
- 存在使用者級別的預算，且 **$0** 的使用者預算意味著無法訪問按量計費的功能

該預算系統是企業、組織、成本中心和使用者範圍內 AI 額度使用的直接支出上限。

重要警告：預算**不會**讓提示詞變小。它們**不會**減少每個提示詞的 Token。它們限制的是 AI 額度的支出。

實踐中的預設做法：

1. 為推廣設置一個非零的測試預算
2. 儘早啟用警報
3. 一旦報告看起來正常，就啟用「停止使用」
4. 每月進行審查，而不僅僅是在超支之後

## 2. 使用使用者級別預算進行每使用者收緊

對於 6 月 1 日之後的 Business 和 Enterprise，最乾淨的每使用者上限是 GitHub [用量計費指南](https://docs.github.com/en/copilot/concepts/billing/usage-based-billing-for-organizations-and-enterprises) 中記錄的**使用者級別 AI 額度預算**。GitHub 文檔明確表示，使用者級別預算為 **$0** 意味著該使用者無法訪問按量計費的功能。

管理員心態：首先思考 AI 額度使用情況。模型選擇、對話深度、代理持續時間和使用者級別預算現在都是直接的支出控制項。

**不要**聲稱 IDE 中的 Copilot Chat 存在已記錄的組織級別每使用者模型切換。GitHub 的文檔描述了預算、成本中心和津貼管理。雖然他們記錄了成員級別的組織和企業模型訪問政策，但並不是一個你可以在一個組織內任意微調每個使用者模型矩陣的控制項。

實踐模式：

1. 基準群組獲得較低的使用者級別 AI 額度預算
2. 高級使用者僅在有明確工作需求時才獲得更多預算
3. 每月審查：降低那些使用情況無法支撐額外成本的使用者的預算

6 月 1 日後的管理員提示：

1. 留意長時間的聊天和代理會話，因為重複的對話輪數現在會直接累積 AI 額度成本
2. 密切監控使用者級別預算，因為 **$0** 會移除按量計費功能的訪問權限
3. 記住，代碼補全和下一步編輯建議不按 AI 額度計費，因此不要將所有 Copilot 活動視為同等成本
4. 定期審查模型選擇，因為聊天式互動現在會根據模型和 Token 消耗 AI 額度
5. 在 IDE 工作流中保持精簡的存儲庫指令，因為即使計費轉向 AI 額度，較低的輸出量仍能減少 AI 額度消耗

遺留說明：如果你仍在 6 月 1 日之前為客戶做準備，[進階請求管理](https://docs.github.com/en/copilot/concepts/billing/premium-request-management) 解釋了即將退場的控制模型。請僅將其作為過渡指導使用。

## 3. 在啟用模型前進行審核

請使用 [配置 Copilot 中的 AI 模型訪問權限](https://docs.github.com/en/copilot/using-github-copilot/ai-models/configuring-access-to-ai-models-in-copilot)。

GitHub 文檔說明組織所有者和企業所有者可以為成員啟用或禁用 AI 模型的訪問權限。請將其視為受支持的模型治理控制項。

這與在單個組織內進行精細調整的每使用者模型切換並不相同。對於不同的組織級別模型政策或分開的計費邊界，分開組織可能仍然有用。僅當邊界值得其開銷時才這樣做。

實踐審核清單：

1. 哪些工作流實際需要進階模型
2. 哪些團隊從中產生了可衡量的價值
3. 哪些使用者可以留在 Auto 或包含在內的模型中
4. 啟用後你將檢查哪些使用報告
5. 如果支出在沒有品質提升的情況下跳升，你將使用什麼回滾規則

糟糕的推廣模式：僅因一小群人要求處理一項艱鉅任務，就為整個組織開啟每個進階模型。

更好的推廣模式：分階段批准模型訪問，然後在擴大範圍前比較使用情況和輸出品質。

## 4. 將自定義指令保持在正確的範圍內

GitHub 在 [為 GitHub Copilot 添加存儲庫自定義指令](https://docs.github.com/en/copilot/how-tos/configure-custom-instructions/add-repository-instructions) 及其連結的 IDE 特定頁面中記錄了 IDE 和 GitHub.com 的存儲庫指令。

GitHub 在 [為 GitHub Copilot 添加組織自定義指令](https://docs.github.com/en/copilot/how-tos/configure-custom-instructions/add-organization-instructions) 中記錄了組織指令。

GitHub 文檔中的重要範圍規則：

- 存儲庫自定義指令是有效的存儲庫級別指導
- 組織自定義指令目前支持 GitHub.com 上的 Copilot Chat、GitHub.com 上的 Copilot 代碼審查以及 GitHub.com 上的 Copilot 雲端代理
- **不要**假設組織指令適用於 IDE

最後一點對成本控制至關重要。如果你希望在 VS Code 或 JetBrains 中看到精簡行為，請將始終開啟的指導放在存儲庫指令中，而不僅僅是組織指令中。

實踐中的拆分：

- 組織指令：GitHub.com 介面的廣泛政策基調、審查提醒、跨存儲庫預設值
- 存儲庫指令：必須套用於 IDE 工作流的具體編碼、構建和精簡規則
- 路徑特定指令：額外上下文值得投入成本的狹窄局部規則

## 5. 分開組織可以運作，但請將其視為有前提的變通方法

GitHub 的模型訪問和計費控制使得分開組織成為維持不同組織級別模型政策或計費邊界的可能方式。

當你需要為不同的使用者群體維護不同的組織級別模型政策或獨特的計費邊界，且你目前的組織結構已經與這些群體匹配時，這會很有用。

但這是一個有前提的變通方法，而非神奇的治理功能。

權衡：

- 更多的管理開銷
- 許可證分配的複雜性
- 如果使用者跨越多個組織，則會顯得很彆扭
- SCIM 和成本中心結構提供的答案可能比僅靠 Copilot 更有說服力

僅當政策或計費邊界值得運營成本時才使用分開組織。這是一個後備方案，而非首選。

## 6. 衡量正確的事物

請使用上述計費文檔中的 GitHub 使用報告和 AI 額度報告。詢問四個問題：

1. 哪些團隊最常超出基準支出
2. 哪些使用者推動了最多的 AI 額度使用
3. 哪些模型與更好的結果（而非僅僅是習慣）相關
4. 哪些代理工作流產生了巨大支出卻沒有相應的交付價值

當對比模型品質和價格狀況時，補充性的基準測試會有所幫助。[llm-stats.com](https://llm-stats.com/) 作為獨立參考點非常有用。請將其視為補充基準，**而非**官方 GitHub 指導，也**不能**替代你自己的 Copilot 使用報告。

## 7. 6 月 1 日切換清單

如果你正在為客戶準備 6 月 1 日的轉變，請先執行以下操作：

1. 將管理指導從請求計數器轉移到 AI 額度預算
2. 決定哪些使用者需要嚴格的使用者級別預算，以及哪些團隊可以共享更廣泛的池化預算
3. 在前沿模型變為直接的 AI 額度支出前審核其可用性
4. 提醒團隊，代碼補全和下一步編輯建議仍不計入 AI 額度計費
5. 首先關注長時間的聊天和代理工作流，因為它們會成為最快的支出放大器

## 推薦的企業預設設置

1. 將 Auto 作為預設模型路徑
2. 在廣泛推廣前設置預算
3. 當你需要更嚴格的每使用者控制時，使用使用者級別的 AI 額度預算
4. 在啟用進階模型前進行審核
5. 保持存儲庫指令簡短，以便 IDE 工作流繼承正確的預設值
6. 僅在成本中心邊界已經支持的情況下使用分開的組織細分

這是刻意設計得乏味的。廉價的預設優先。進階訪問按例外處理。擴大前先測量。
