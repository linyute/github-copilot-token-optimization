快速上手

現在就做的 11 件事

從輸出開始，然後縮減結構化輸入，最後路由模型與工具。

最快獲勝的方法

  1. 僅回覆程式碼：每 Token 投資報酬率最高。
  2. 約束格式：項目符號、單句、JSON、表格。
  3. 縮減常駐上下文：指令及 Agent 文件。
  4. 預設使用 Auto (自動)：僅在必要時指定進階模型。
  5. 簡單問題請使用 Ask (詢問) 模式。

更多快速獲勝的方法

  1. 使用 applyTo 限定上下文範圍。
  2. 撰寫精確提示詞：指定文件、函式、完成條件。
  3. 針對目標模型調整提示詞。
  4. 審核 MCP 伺服器。
  5. 在 Copilot CLI 中執行 /chronicle improve。
  6. 長鏈 CLI 工具嘗試使用 CodeAct。

優先順序堆疊

輸出第一

冗長的回答很昂貴。一條預設指令就能改變每次回應。

常駐第二

指令文件與工具架構會重複載入。微小的基數會產生複利效應。

模式第三

Ask 用於提問。Edit 用於單一文件變更。Agent 用於多步驟工作。

治理第四

預算設定支出上限。模型策略限制昂貴的路徑。

實驗 0：建立習慣基準

目標

在學習技巧前，找出你前 3 名的 Token 洩漏點。

動作：將以下各項標註為綠/黃/紅：輸出冗長、指令大小、開啟的分頁、Agent 使用頻率、MCP 數量、固定模型、長對話歷史。

產出：個人基準 + 一項「今日修正」行動。

時間：10 分鐘。

第 1 部分

為何 Token 很重要

Token 驅動了成本、速度、限制及上下文容量。

Token 是次單字 (Subwords)

BPE 行為

  • 常見單字可以是單個 Token。
  • 罕見單字會被拆分。
  • 標點符號與贅字仍然計費。

核心洞見

Sure, I'd be happy to help! 可能會消耗約 10 個完全沒有技術價值的 Token。

成本模型

類別內容控制槓桿
輸入 (Input)提示詞、上下文、文件、歷史、工具範圍限定與壓縮
快取 (Cached)穩定且重複使用的前綴保持穩定上下文的穩定性
輸出 (Output)模型回應僅程式碼、簡潔格式

在供應商的定價範例中，輸出 Token 的成本通常顯著高於輸入 Token。

輸入比你的提示詞還大

上下文視窗 (Context window)
├─ 系統提示詞 (System prompt)
├─ copilot-instructions.md / AGENTS.md
├─ 文件內容與開啟的分頁
├─ 對話歷史
├─ MCP 工具架構 (Schemas)
└─ 你輸入的提示詞

你 20 個字的提示詞可能夾帶在數千個隱藏 Token 中。

Coding Agent 乘數效應

從 Issue 到 PR 的過程並非單次請求。

Agent 步驟

規劃、搜尋、讀取文件、編輯、執行測試、檢查失敗、重試。

複利效應

每一步都會重新載入基準上下文，並加上先前的工具執行結果。

目標

區分可見的提示詞與隱藏的上下文。

動作：選擇一個最近的 Copilot 任務。列出可能的上下文來源：開啟的文件、引用的文件、指令、歷史紀錄、工具、輸出。

討論：哪個來源最容易控制？

第 2.1 部分

提示詞壓縮

用更少的 Token 表達相同內容。保留技術精確度。

穴居人 (Caveman) 模式

丟棄

冠詞、填充詞、客套話、模稜兩可的詞、修飾語。

保留

技術術語、程式碼、檔名、約束條件、確切的完成條件。

[事物] [動作] [原因]. [下一步].

調整前後對照

冗長穴居人模式
能否請你審閱一下這個 pull request 並告訴我是否有任何問題？ 審閱 PR。標記問題。
你能解釋一下這個錯誤訊息的意思以及我該如何修復它嗎？ 解釋錯誤。如何修復。

壓縮等級

等級風格適用時機
輕量 (Lite)專業但緊湊面對客戶、引導新手
完全 (Full)碎片化，無廢話日常開發者提示詞
極致 (Ultra)縮寫、箭頭、極簡高頻率、已知領域

結構化優於散文

POST /api/users
驗證：
- name: string, 必填
- email: string, 必填, 有效格式
驗證失敗回傳 400
成功回傳 201, 並回傳建立的 user
儲存至 DB

項目符號與鍵值對強迫釐清需求並減少贅字。

以程式碼為中心的提示

自然語言

建立一個函式，接收數字，過濾負值，將剩餘值加倍，回傳總和。

程式碼中心

fn(nums) -> filter(>0) -> map(*2) -> sum

類似 getUserById 但改用 email。若缺失回傳 404。

將防護欄視為不變量 (Invariants)

冗長程序壓縮後的不變量
確保每個 SQL 查詢都使用參數化數值...SQL：僅限參數化查詢。禁止字串拼接。
請為任何新程式碼編寫測試...新程式碼 -> 測試。涵蓋成功 + 錯誤案例。

目標

將一個真實提示詞用三種方式重寫：輕量、完全、極致。

動作：保留檔名、行號、約束、完成條件。僅丟棄廢話。

比較：哪個版本最短且不增加歧義？

實驗 3：提示詞 A/B 測試

試試這個 vs 那個

A: 你能看一下 auth 並改善錯誤處理嗎？

B: 檔案: src/auth/login.ts.
錯誤: user 為 null 會導致 .email 崩潰。
修正: 在 .email 前增加 null 檢查。
測試: 增加 user 為 null 的案例。
完成: 定向測試通過。

執行：分別用 A 和 B 詢問 Copilot。比較文件讀取次數、釐清問題次數、輸出長度。

第 2.2 部分

語言比較

對於提示詞，英文通常是 Token 效率最高的語言。

CJK 假設的破滅

語言句子Token 數vs 英文
英文 (English)I met a huge dog51.0x
西班牙文 (Spanish)Conoci a un perro enorme81.6x
中文 (Chinese)我遇到一隻巨大的狗112.2x
日文 (Japanese)巨大な犬に出会った112.2x
俄文 (Russian)Я встретил огромную собаку142.8x

為什麼

  • BPE 分詞器 (Tokenizers) 大量基於英文訓練。
  • 常見英文單字通常僅佔 1 個 Token。
  • CJK 文本雖然字數較少，但每個字的平均 Token 數較多。
  • 非拉丁語系腳本成本通常較高；音譯有時會有幫助。

實務原則

提示詞與指令請使用簡潔的英文。不要為了節省 Token 而翻譯成 CJK。

如果使用者用母語撰寫能有更好的品質，則品質重要性可能超過 Token 成本。但以最佳化而言，英文勝出。

實驗 4：分詞器現實檢查

目標

用實際的 Token 計數來驗證語言假設。

動作：挑選一個提示詞。翻譯成你最擅長的非英語語言。在分詞器工具中比較 Token 計數。

產出：團隊語言指引：何時必須用英文，何時可以用母語。

時間：20 分鐘。

第 2.3 部分

上下文管理

控制發送的內容。結構化輸入是最大的獲勝點。

常駐文件每回合都在計費

.github/copilot-instructions.md, AGENTS.md, CLAUDE.md 等文件可能變成持續存在的上下文。

當多個工具載入多個文件時，重複的內容可能會被重複計費。

指令壓縮範例

簡潔如穴居人。技術內容精確。僅刪廢話。
丟棄：冠詞、填充詞 (just/really/basically)、客套話、修飾語。
可使用碎片句。短同義詞。程式碼不變。
模式：[事物] [動作] [原因]. [下一步].

約 50 個 Token。每次互動都會載入。

上下文衛生

  • 關閉與當前任務無關的文件。
  • 保持文件專注且精簡。
  • 忽略建置產出 (Build output)、套件目錄 (Vendor dirs)、生成的文件。
  • Business/Enterprise 版請對敏感路徑使用內容排除 (Content Exclusion)。
  • 主題變更時啟動新的對話。

使用限定範圍的指令

---
applyTo: "src/api/**/*.ts"
---
API 規範：
- 路由位於 src/api/routes/。Handler 保持精簡，邏輯位於 services/。
- 使用 zod 驗證。
- 透過 Result<T,E> 傳遞錯誤，禁止 throw。

僅在相關文件處於範圍內時，才支付特定路徑指引的成本。

常駐 vs 條件式 vs 隨選

範圍用途Token 策略
常駐 (Always-on)通用規則極致精簡
條件式 (Conditional)路徑/層級規則使用 applyTo
隨選 (On-demand)審閱檢核表、版本說明僅在呼叫時載入

快取友善的行為

  • 將穩定的上下文放在頂部。
  • 不要頻繁改動指令文件。
  • 重複使用儲存的程式碼片段 (Snippets)，而非重新貼上大量上下文。
  • 當歷史紀錄陳舊時，總結並重啟對話。

企業備註：內容排除 (Content Exclusion)

客戶端行動

針對敏感檔案、生成的 Bundle、大型數據檔案及受監管的路徑配置內容排除。

重要：將此視為隱私/策略控制為首要，Token 節省為次要副作用。請參考官方文件檢查當前支援範圍。

實驗 5：上下文審核

目標

找出應改為限定範圍或隨選載入的常駐上下文。

動作：檢查儲存庫指令、AGENTS/CLAUDE 文件、開啟的分頁、生成的檔案、MCP 配置。

產出：三欄清單：保留常駐、移動至 applyTo、移動至隨選/刪除。

第 2.4 部分

輸出控制

告訴模型「不要」說什麼。每 Token 投資報酬率最高。

預設常用指令

僅回覆程式碼，不要解釋。

當團隊已理解所需變更時，適用於生成任務。

格式約束

指令適用時機
用一句話回答快速決定或解釋
最多 3 個項目符號便於掃視的總結
以 JSON 回覆機器可讀的資料擷取
先回覆 yes/no，再用一行解釋原因審核閘口
僅顯示 Diff修補程式 (Patch) 檢查

專案預設

保持簡潔。除非要求，否則不予解釋。
生成任務僅提供程式碼。
項目符號優於段落。

在學習、除錯或教學時覆寫：明確要求解釋。

實驗 6：輸出 A/B 測試

試試這個 vs 那個

A: 為 processOrder() 增加輸入驗證。

B: 為 processOrder() 增加輸入驗證。
僅回覆程式碼。不要解釋。極小化 Diff。

衡量：回應長度、解釋用的 Token、Diff 的實用性。

時間：15 分鐘。

預設使用 Auto 模型

Auto 是一個良好的基準：支援 Auto 池、具備 Org 策略意識，且根據文件，符合條件的付費方案 Chat 使用可享有折扣。

Auto 並非指「自動升級到每個進階模型」。請有意識地指定進階模型。

目標模型：GPT-5.5.

第 2.6 部分

常駐上下文問題

過多的上下文可能讓 Agent 表現變差且成本更高。

研究訊號

發現觀察到的影響
LLM 生成的上下文文件在 5/8 的設定中有負面影響
平均正確性下降約 2%
Token 成本增加 20-23%
推理開銷在引用的設定中增加約 22%

為何上下文會有負面影響

冗餘稅

重複 Agent 即可發現的事實。

注意力稅

重要規則淹沒在中間。

錨定陷阱

過時的工具指引過度影響 Agent。

訊號/雜訊

例行公事沖淡了真正的地雷。

僅保留「地雷」 (Landmines)

保留刪除
使用 uv 取代 pip這是一個 Python 專案

Bug 追蹤器模型

起步時幾乎保持空白。
Agent 出錯時 -> 增加一行規則。
根因修正後 -> 刪除該行規則。

指令文件應該增長與縮減，而非像 Wiki 一樣不斷累積。

目標

在不損失正確性的情況下削減上下文。

動作：將每條指令標記為：可發現、風格、地雷、過時、重複。刪除或限定除真正常駐地雷以外的所有內容。

產出：最多 10 行的指令文件草案。

時間：35 分鐘。

第 2.7 部分

MCP 與工具成本

工具架構是隱藏的 Token 稅。

先衡量

/context

系統/工具: MCPs + 指令 + 系統提示詞
訊息: 對話歷史
剩餘空間: 剩餘上下文容量

在 VS Code 中，估計為：啟用的 MCP 伺服器 x 工具數 x 約 200 Token/工具。

每個工具都要 Token 成本

組成部分預估成本
名稱 + 描述20-50 Token
簡單參數架構30-80 Token
複雜參數架構100-300 Token
每個工具總計100-500 Token

相乘問題

10 個 MCP 伺服器
x 每個伺服器 5 個工具
x 200 Token/工具
= 每一步 10,000 Token

15 個 Agent 步驟 = 150,000 架構 Token

審核原則

  • 停用當前不需要的伺服器。
  • 使用個別工作區配置，而非全域開啟所有工具。
  • 優先使用內建工具，而非重複的檔案系統 MCP。
  • 將 Skills 用於偶發的功能需求。
  • 若支援，按命名空間/模式限定大型外掛程式範圍。

工具輸出壓縮

RTK

CLI 代理，在 Agent 讀取前過濾吵雜的 Shell 輸出：測試、git diff、grep、日誌、檔案列表。

rtk init --copilot

Azure MCP 案例模式

一個大型外掛程式可能佔據大部分「系統/工具」預算。

--namespace appservice --namespace cosmos --namespace keyvault --namespace storage

按服務/角色 (Persona) 限定範圍。除非真的需要，否則避免「所有工具」模式。

實驗 11：MCP 審核

目標

移除隱藏的架構開銷。

第 4.2 部分

模型選擇與定價

區分 Copilot 文件、UBB 框架及供應商 Token 定價。

三種定價視角

視角解答的問題
GitHub Copilot 文件方案可用性、模型存取、Auto 行為、發佈的訊號
UBB 框架切換後的 AI 點數預算與治理
供應商 API 定價輸入 vs 輸出的直覺認知，而非 Copilot 計費表

Auto (自動)：實際意義

  • 從支援的 Auto 池中選擇。
  • 受限於方案與 Org 策略。
  • 基於健康狀況與效能。
  • 根據文件，符合條件的付費方案 Chat 使用可享折扣。
  • 不會自動升級到每個進階模型。

推理努力程度 (Reasoning effort)

努力程度用途
低 (Low)高頻率的簡單對話/分類
中 (Medium)典型的程式編寫與重度依賴工具的工作 (若支援)
高/極高 (High/max)架構、安全、新穎的拆解任務

僅在有公開此控制項的模型上使用。

反模式

  • 整天固定在進階模型上。
  • 假設當任務變難時 Auto 會跳轉到進階模型。
  • 將供應商 API 定價等同於 Copilot 計費。
  • 優先為整個 Org 開啟所有進階模型。
  • 忽略方案/模型可用性。

實驗 14：模型升級策略

定義使用者何時可以指定進階模型，以及何時必須使用 Auto。

動作：將工作流映射至預設模型路徑、觸發升級的條件、預期價值、回退訊號。

產出：單頁模型策略草案。

時間：30 分鐘。

第 4.3 部分

企業治理

管理控制項設定支出上限。提示詞習慣提高效率。

支出槓桿

  1. 預算天花板。
  2. 使用者層級的 AI 點數預算。
  3. 模型可用性策略。

提示詞壓縮並非限制支出，而是減少核准使用量內的浪費。

先設定預算

客戶端行動

設定企業/Org/成本中心預算。儘早開啟警示。在報表可信後開啟停止使用 (Stop-usage)。每月審查。

使用者層級的緊縮

  • 基準使用者：低 AI 點數預算。
  • 權力使用者 (Power users)：基於工作需求給予較高預算。
  • $0 預算：禁止使用基於使用量計費的功能。
  • 每月審查並調降未使用的配置。

啟用前先審查模型

問題決定
哪些工作流需要進階模型？精確限定範圍地開啟
哪些團隊能創造可衡量的價值？分配較高預算
哪些使用者可以留在 Auto 路徑？保持預設路徑的廉價
回退訊號是什麼？使用量增加，但價值持平

指令範圍治理

  • Org 指令：寬泛的 GitHub.com 策略與審核提醒。
  • 儲存庫指令：IDE 工作流預設值與編碼規則。
  • 特定路徑指令：付費上下文中的本地規則。
  • 不要假設 Org 指令適用於所有地方；請檢查文件。

獨立 Org：僅作為備援

適用於 Org 結構已匹配群組時的不同模型策略或計費邊界。

成本：管理開銷、授權複雜度、使用者分散、SCIM/成本中心約束。

衡量正確的事物

  1. 哪些團隊超出基準？
  2. 哪些使用者驅動了 AI 點數使用？
  3. 哪些模型改善了成果？
  4. 哪些 Agent 工作流在沒有交付價值的狀況下產生支出？

6 月 1 日切換檢核表

  1. 從請求計數器轉換為 AI 點數預算。
  2. 決定資金池模式 vs 使用者層級預算。
  3. 在進階模型變成直接支出前審核其可用性。
  4. 提醒團隊：根據文件，自動補全/下次編輯不計入 AI 點數帳單。
  5. 首先關注長對話與 Agent 工作流。
